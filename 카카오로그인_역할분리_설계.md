# 카카오 로그인 + 역할 분리(환자/의사) 설계

> 목표: **카카오 회원가입(필수 로그인)** 후, 계정 역할에 따라
> **환자 → 채팅 앱(app.html)** / **의사 → 환자 리포트 대시보드(doctor.html)** 로 갈라진다.
> 방식: Supabase Auth의 Kakao OAuth 공급자(별도 SDK 불필요). 이 문서는 Claude Code 구현용.

---

## 0. 원칙 준수 (CLAUDE.md)
- **진단 안 함** / **위험감지 규칙 기반 유지(`RISK_WORDS`)** — 그대로.
- **Solar 키는 백엔드에만** — 그대로. (카카오 키는 Supabase/카카오 콘솔에 둠, 프론트엔 anon key만.)
- **데모 안 깨짐** — 필수 로그인이지만, 로그인/네트워크 실패가 시연을 막지 않게 **데모 안전장치**(7절) 둔다.
- 로그인 도입으로 **실제 사용자 데이터가 가능**해지므로 RLS를 **실잠금**(5절)으로 전환 — 더는 "아무나 읽기" 아님.

## 1. 전체 흐름
```
앱 진입 → 세션 확인
  ├─ 세션 없음 → [로그인 화면] → "카카오로 시작하기" → Kakao OAuth → 콜백 복귀
  └─ 세션 있음 → profiles.role 조회
        ├─ 'patient' → 환자 채팅(app.html 본문)  · entries.patient_id = 내 user.id
        └─ 'doctor'  → 의사 대시보드(doctor.html) · 환자 목록 → 선택 → 리포트
```

---

## 2. 결정 사항 (데모 기준)
- **필수 로그인**: 로그인 전엔 채팅/대시보드 접근 불가.
- **역할 2종**: `patient`(기본), `doctor`.
- **역할 부여(데모)**: 의사 계정은 Supabase에서 `profiles.role`을 수동으로 `doctor`로 변경.
  (운영 전환 시 초대코드/도메인 기반 자동 부여로 업그레이드 — 10절.)
- **의사↔환자 연결(데모)**: 의사는 **모든 환자**를 조회(환자 목록에서 선택).
  (운영 전환 시 `care_links`로 담당 환자만 — 10절.)

## 5. DB 스키마 + RLS 실잠금 (Supabase SQL Editor)

> 2026-06-19 보안 보강: 아래 초기 스키마를 실행한 프로젝트는
> `SUPABASE_보안강화_20260619.sql`도 이어서 실행한다. 이 파일은 사용자의
> `role` 직접 변경을 차단하고, 의사 조회 시 환자의 공유 설정을 DB에서 강제한다.

```sql
-- 1) 프로필(역할) 테이블
create table if not exists profiles (
  id           uuid primary key references auth.users(id) on delete cascade,
  role         text not null default 'patient' check (role in ('patient','doctor')),
  display_name text,
  created_at   timestamptz not null default now()
);

-- 2) 재귀 RLS 방지용 역할 확인 함수(security definer = RLS 우회로 안전)
create or replace function is_doctor() returns boolean
  language sql security definer stable as $$
  select exists(select 1 from profiles where id = auth.uid() and role = 'doctor');
$$;

-- 3) 신규 가입 시 프로필 자동 생성
create or replace function handle_new_user() returns trigger
  language plpgsql security definer as $$
begin
  insert into public.profiles (id, display_name)
  values (new.id, new.raw_user_meta_data->>'name')
  on conflict (id) do nothing;
  return new;
end; $$;
drop trigger if exists on_auth_user_created on auth.users;
create trigger on_auth_user_created after insert on auth.users
  for each row execute function handle_new_user();

-- 4) profiles RLS
alter table profiles enable row level security;
create policy "own profile select" on profiles for select using (auth.uid() = id);
create policy "own profile update" on profiles for update using (auth.uid() = id);
create policy "doctor reads profiles" on profiles for select using (is_doctor());

-- 5) entries RLS 교체: 데모(아무나) → 인증 기반
drop policy if exists demo_insert on entries;
drop policy if exists demo_select on entries;
-- 환자: 본인 행만 (patient_id = 내 uid를 text로 저장)
create policy "patient insert own" on entries for insert
  with check (patient_id = auth.uid()::text);
create policy "patient select own" on entries for select
  using (patient_id = auth.uid()::text);
-- 의사: 모든 환자 행 조회(데모)
create policy "doctor select all entries" on entries for select using (is_doctor());
```
> 주의: `entries.patient_id`는 현재 text. 앞으로 **로그인 사용자의 `user.id`(uuid)를 text로 저장**한다.
> 기존 `demo-xxxx` 데이터는 인증 정책과 안 맞아 안 보이게 됨 → 발표용으로 새 계정으로 다시 쌓으면 됨.

## 6. 프론트 구현 (Claude Code)

### 6-1. app.html — 로그인 게이트 + 역할 분기
- 진입 시: `sb.auth.getSession()` → 없으면 **로그인 화면**(카카오 버튼만) 표시, 채팅 UI 숨김.
- 로그인 버튼: `sb.auth.signInWithOAuth({ provider:'kakao', options:{ redirectTo: location.href }})`.
- 콜백 복귀 후 세션 생기면: `profiles.role` 조회 →
  - `doctor` → `location.href = 'doctor.html'`
  - `patient` → 채팅 UI 노출, **`patientId = user.id`** 로 설정(localStorage 임의 id 대신 사용).
- 로그아웃 버튼(계정 메뉴): `sb.auth.signOut()` → 로그인 화면.
- `saveSignalToCloud`/`loadCloudEntries`는 그대로 쓰되 `patient_id`에 `user.id` 사용.

```js
async function boot(){
  if(!sb){ return showLoginError(); }
  const { data:{ session } } = await sb.auth.getSession();
  if(!session) return showLogin();
  const u = session.user;
  const { data:prof } = await sb.from('profiles').select('role').eq('id', u.id).single();
  if(prof?.role === 'doctor'){ location.href = 'doctor.html'; return; }
  patientId = u.id;                                    // 환자 식별 = 계정
  initPatientApp();
}
sb && sb.auth.onAuthStateChange(()=> boot());
boot();
```

### 6-2. doctor.html — 의사 대시보드 (신규 파일)
- 진입 시 동일 게이트: 미로그인 → 로그인, 로그인했는데 `role!=='doctor'` → "권한 없음" 또는 app.html로.
- 환자 목록: `profiles.select('id,display_name').eq('role','patient')` (의사 RLS로 허용).
- 환자 선택 → `loadCloudEntries(pid)` → 기존 `aggregate()`/리포트 렌더 재사용(app.html의 리포트 로직 이식).
- 의사 화면은 **읽기 전용**(환자 통제 토글 없음, 예외 Red Flag만 부각 — 의사 추가 업무 0 원칙).

## 7. 데모와 로그인 원칙
- **콜드스타트/세션**: 발표 전 카카오 로그인 1회 미리 해두면 세션 유지됨(다시 안 물음).
- **환자 대화 로그인 필수**: `app.html`은 Supabase 세션이 없으면 채팅을 노출하거나 전송하지 않는다. `/chat`과 `/extract`도 Bearer 토큰을 검증한다.
- **무로그인 평가 경로**: 실제 대화 대신 `index.html`의 가상 자동 시나리오로 환자 대화→리포트 흐름을 보여준다.
- **계정 미리 준비**: 환자용 카카오 계정 1개 + 의사용 1개(또는 같은 계정 role만 바꿔가며) 사전 로그인 테스트.

## 9. 테스트 체크리스트
- [ ] 미로그인 상태에서 채팅/대시보드 접근 불가(로그인 화면만).
- [ ] 카카오 로그인 성공 → 환자는 채팅, 의사는 doctor.html.
- [ ] 환자가 보낸 메시지가 `entries`에 `patient_id = 내 user.id`로 저장.
- [ ] 환자는 **남의 데이터 조회 불가**(RLS), 의사는 환자 목록·리포트 조회 가능.
- [ ] `red_flag`는 여전히 프론트 규칙 기반.
- [ ] `?demo=1` 우회로 로그인 없이 로컬 데모 동작(비상구).
- [ ] 로그아웃 → 다시 로그인 화면.

## 10. 운영 전환 시 (발표엔 불필요)
