# BUILD SPEC — 챗봇형 환자 앱 (v2)

> Claude Code 작업 지시서. CLAUDE.md 원칙을 먼저 읽고, 이 스펙대로 만든다.
> 와이어프레임: 21팀_와이어프레임.html (GPT 스타일 챗봇). 핵심 = "겉은 일반 챗봇, 임상 수집은 대화 속 비가시적".

## ⚠️ 가장 먼저 (안전)
- **`index.html`(현재 배포된 가발표 데모)을 절대 덮어쓰지 말 것.**
- 새 파일 **`app.html`**로 v2를 만든다. v2 완성·테스트 후 사람이 교체 결정.
- 작게 커밋하며 진행. 키·.env 커밋 금지.

## 목표
환자 앱을 일반 챗봇(GPT/Claude 모바일 스타일)처럼 만든다. 폼·체크리스트·슬라이더 없음.
환자는 그냥 수다 떠는데, 뒤에서 임상 신호가 조용히 추출되고, 리포트는 계정 메뉴 안에만 있다.

## 핵심 아키텍처
1. **매 메시지 = 진짜 범용 LLM 답변.** 코딩 질문이든 일상이든 GPT/Claude처럼 실제로 답한다(정해진 답 금지). → 백엔드(Render 프록시)에 채팅 엔드포인트 추가 필요(`/chat`), Solar 호출.
2. **백그라운드 임상 추출.** 답변과 별개로, 그 메시지에서 수면·약물·기분·스트레스원을 조용히 추출(기존 `/extract` 재사용). 화면엔 "조용히 추출" 같은 개발용 표시 없이, 사용자에겐 안 보이게.
3. **위험 감지 = 규칙 기반(프론트, RISK_WORDS).** 항상 작동. API 의존 금지.
4. **데모 폴백.** `BACKEND_URL` 비었거나 실패 시: 채팅은 간단한 목업 응답, 추출은 기존 규칙 기반 목업. 데모가 안 깨지게(5초 타임아웃 유지).
5. 온보딩 고지 1회(첫 AI 메시지 또는 카드): "나는 AI예요(의사·치료사 아님). 편하게 얘기하면 진료에 도움되게 정리해 선생님께만 전해요. 무엇을 전할지는 직접 골라요. 위험한 상황이 감지되면 안전을 위해 선생님께 알림이 갈 수 있어요." 이후 반복 안 함.

## 화면 (와이어프레임 5개 그대로)
**① 메인 채팅** — 햄버거 / 앱명 / 검색 / 아바타. 말풍선 대화. 입력창 "무엇이든 이야기해 보세요". 폼 없음.
**② 사이드 메뉴(☰)** — ＋새 대화, 프로젝트·앱·라이브러리·이미지(겉모양), 대화 기록 목록(클릭 시 이어가기).
**③ 계정 메뉴(아바타)** — 리포트 기록 보기 / 내보내기 / 설정. (임상 기능은 여기에만 숨김)
**④ 위기 자원** — 채팅 말풍선 안 카드(모달·화면전환 없음): 자살예방 109(24시간·무료), 정신건강 위기상담 1577-0199, 담당 병원 연락처. 공감 응답 + 카드, 대화 계속 이어짐. 리포트에 우선 플래그.
**⑤ 리포트(계정 메뉴 → 리포트 기록 보기)** — 5영역(약물 순응도 / 수면·기능 / 안전·리스크 / 스트레스원·정신상태) + "의사에게 보여주기(전체화면)" + "PDF 내보내기" + disclaimer("진단이 아닌 대화 기록 요약, 의학 판단은 의사가").
  - **추가:** 리포트가 의사에게 가기 전, 환자가 **항목별 '비공개' 선택** 가능(우리 통제권 설계). 단, 안전 Red Flag는 항상 포함(빼기 불가) — 이건 온보딩에서 고지됨.

## 설정(Settings) — "일반 챗봇과 다른 부분은 여기"
- 외래 일정(다음 진료일) 입력
- 담당 병원·연락처(위기 카드에 쓰임)
- 처방 약물(모니터링 항목 근거 — 선택)
- 리포트 공유 기본값/비공개 설정
- 온보딩 고지 다시 보기

## 현재 코드에서 가져올 것 (재사용)
- `RISK_WORDS`, `getSignals`의 목업 폴백·5초 타임아웃, `makeReport`의 5영역 구조.

## 기술
- 단일 HTML(프레임워크 없이) 권장 — 데모 안정성·배포 단순. 팀 판단으로 React 전환 가능하나 지금은 단일 파일 유지가 안전.
- LLM: Solar(Upstage) via Render 프록시. 키는 Render 환경변수.
- (선택) Supabase 저장은 이후 단계.

## 의사 대시보드 (Phase 2 — 지금은 안 만듦)
환자주도(폰/PDF) 방식이 MVP라 의사 대시보드는 당장 불필요. 나중에 별도 웹으로:
- 최상단 안전 **Triage 배너**(Red Flag) → 시계열 그래프(수면·복약·기분) → 원문 drill-down.
- 예외 기반(별일 없으면 한 줄), 진료 전 10초 파악, **의사 추가 업무 0**.
- 의사 입력은 처방·일정만(선택·생략 가능).

## 작업 순서 제안 (Claude Code)
1. `index.html` 복사 → `app.html` 시작점.
2. UI를 챗봇형으로(①②③). 사이드·계정 메뉴.
3. `/chat` 백엔드 추가 + 프론트 연결(범용 답변). 폴백 포함.
4. 백그라운드 추출 연결(기존 `/extract`).

7. 설정 화면.
각 단계 후 사람이 확인. 한 번에 다 하지 말 것.

## /chat 시스템 프롬프트 (작업순서 3 — 이걸 그대로 system 메시지로)
> 설계 철학: "금지 규칙 50개"가 아니라 **핵심 원칙 + few-shot 예시**. 단, 안전선은 별도로 딱딱하게.
> 두 층 — ① 스타일(원칙+예시, 자연스럽게) ② 안전선(예외 없음). 안전선을 자연스러움으로 녹이지 말 것.

```
너는 사용자가 편하게 이야기하는 대화형 AI다.
ChatGPT처럼 자연스럽게 답하고, 상담사처럼 교과서적으로 말하지 않는다.

[기본]
- 사용자의 말투·에너지를 따라간다. 가볍게 말하면 가볍게, 진지하면 진지하게.
- 무조건 맞장구치지 않는다. 공감하되 판단은 솔직하게.
- 짧고 담백하게. 설교·번호목록·"~해보세요" 식 조언은 안 한다.
- 코딩이든 잡담이든 실제로 도움이 되게 답한다. 모르면 모른다고 한다.

[딱 이건 지켜 — 가볍게, 친구처럼 (규칙 나열 X)]
- 너는 친구지 의사가 아니야. 진단하거나 "약 이렇게 해" 같은 말은 안 해. 그런 건 "그건 선생님이랑 얘기해봐"로 자연스럽게 넘겨.
- 정말 위험해 보이면(죽고 싶다 등) 가르치려 들지 말고, 그 마음을 먼저 따뜻하게 받아준 다음 곁을 내줘 — "혼자 두기 싫어, 자살예방상담 109도 있어" 정도로. 차갑게 끊지 마.
- 자해하는 법은 알려주지 마.

[예시]
사용자: 배고파
좋음: 그럼 뭐 좀 먹자. 지금 당기는 거 있어?
나쁨: 배고픔은 신체 에너지 부족 신호입니다.

사용자: 오늘 힘들었어
좋음: 오늘 꽤 빡센 날이었나 보네. 무슨 일 있었어?
나쁨: 힘들 때는 심호흡을 하고 감정을 탐색해보세요.

사용자: 리액트 useState 자꾸 꼬여
좋음: 아 그거 흔해 ㅋㅋ 어디서 꼬였어? 코드 보여줄래?
나쁨: 지금은 그 얘기보다 마음 상태를 살펴볼까요?

사용자: 요즘 다 사라졌으면 좋겠어
좋음: 그런 생각이 들 만큼 힘들었구나. 말해줘서 고마워. 혼자 두고 싶지 않아 —
      지금 바로 얘기 나눌 수 있는 곳도 있어(자살예방상담 109). 무슨 일이 있었는지 더 들려줄래?
나쁨: 그런 생각은 위험합니다. 전문가와 상담하세요. / 힘내요, 다 잘 될 거예요.
```

> MVP 이후: 좋은/나쁜 답변 50~100개 모아 A/B로 "마음기록 대화 스타일 가이드"로 발전. 지금은 위 예시면 충분.

## 작업순서 3-B: A/B 선호 수집 (스타일 학습 데이터)
> 목표: GPT처럼 가끔 답변 2개를 보여주고 사용자가 더 좋은 쪽을 고르게 해서 '좋은 답변' 데이터를 모은다.
> ⚠️ **선호 데이터 수집**이지 실시간 RLHF 학습이 아니다(호스티드 Solar는 RL 재학습 불가). 모은 데이터로 프롬프트·few-shot을 개선, 나중에 자체 모델이면 파인튜닝.

**프롬프트:** 위 `/chat` 시스템 프롬프트에서 [태도] 층은 더 가볍게 둬도 됨(말투 따라가기·자연스럽게). **단 [안전선] 블록은 절대 그대로 유지** — 선호와 무관하게 항상.

**A/B UI 설계:**
- 매 턴 2개는 피곤하니 **가끔/선택적**으로: "비교 모드" 토글 또는 N턴마다 1회.
- 비교 시: 같은 입력으로 답변 2개 생성(temperature 다르게 2회 호출 또는 변형 프롬프트) → 나란히 표시 + "어느 답변이 더 좋아요?".
- 고른 답변이 대화에 채택되고, 선택을 기록: `{ ts, user_text, reply_a, reply_b, chosen }`.
- 저장: Supabase `preferences` 테이블(있으면) / 없으면 메모리 + 내보내기. 데모는 가상 데이터.
- **⚠️ 위험 신호가 감지된 응답은 A/B 비교 대상에서 제외.** 위기 응답은 선호로 최적화하지 않고 항상 [안전선]대로. (위험을 '좋아요'로 학습시키면 안 됨)

**나중(MVP 이후):** 모인 선택 데이터에서 '이긴 답변'을 few-shot 예시 라이브러리로 → 프롬프트 개선. 충분히 쌓이고 자체 모델이면 파인튜닝.

- 백엔드 `/chat`: 위 system + 최근 대화(messages 배열)를 Solar에 보내고, 답변 텍스트만 반환.
- 프론트 app.html: `mockReply()`를 `/chat` 호출로 교체. **5초 타임아웃·실패 시 mockReply 폴백** 유지(데모 안 깨짐).
- 위험 감지(RISK_WORDS)·백그라운드 추출은 이 단계에서 건드리지 말 것(각각 5·4단계).
